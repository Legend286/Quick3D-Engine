# Engine Spec — Source-Inspired Game Engine

_Captured after interview rounds with the user. This is the source of truth for scope, architecture, and priorities. No code is written yet._

---

## 0. One-paragraph summary

A C-backed game engine inspired by Valve's Source Engine, with a C# userspace for gameplay, editor tools, and renderer scripting. The C-side hosts a Render Hardware Interface (RHI) targeting **Metal first**, **Vulkan later**. The C#-side hosts FLECS-driven gameplay, a forward+ renderer, scene serialization, model/material/physics asset pipelines, and an Avalonia+ImGui editor suite. Familiar Source-style feel is preserved through folder-organised asset pipelines with tag-versioned binary formats, JSON scene files referencing compiled assets, sidecar meta files (colliders, lightmap UVs). Hot-reloadable C# scripts can attach to entities as components or run as global systems with full engine + RHI access.

## 1. Confirmed architectural decisions (from round 1–4 interview)

| Topic | Decision |
| --- | --- |
| Build system | **CMake** (C/C++ side) + `dotnet build` (C# side). CMake invokes `dotnet build` for assembly outputs and references. |
| FLECS bridge | **Flecs.NET** community bindings for high-level gameplay + custom thin C-export layer for engine-internal systems. Hybrid. |
| C# hot-reload | **AssemblyLoadContext swap**. Compile user code into a DLL via `dotnet build`; unload the old ALC and load the new one. Persistent handles (generational indices) keep state valid. |
| Memory model | **Frame allocator + tiered bump allocators**. One bump allocator per subsystem (renderer, physics, audio). Generational handles for lifetime-stable references. |
| Render pipeline | **Mesh-shader driven Forward+**. Tiled/clustered light culling, single pass for opaques, separate pass for transparency. |
| Anti-aliasing | **MSAA 4× + SMAA post**. No temporal artifacts, well-suited to forward+. Quality preserved on rough materials. |
| Global illumination | **Baked lightmaps** generated by xatlas + offline light baker. Dynamic probes added later. |
| Color pipeline | **Linear HDR + ACES filmic tonemapping**. Industry-standard HDR pipeline. |
| Thread topology | **Game Thread + Render Thread + Physics Thread + Asset IO Thread + ECS worker pool**. Unreal-style progress bar in editor for asset IO. |
| Asset versioning | **Tag-format (dual backends)**. Encode assets using tag versions (basis / compressed / hashed) — engine understands all of them. Engine reads whatever tag the file advertises. |
| C# scripting depth | **Full engine API including RHI** exposed to C#. Hot-reload can swap the rendering DLL too. |
| Editor MVP1 scope | **Material editor + Model viewer + Level editor + Particle editor + ImGuizmo gizmos**. Full editor suite in MVP1. |
| Editor polish layer | **All**: docking + theming, asset thumbnails with offscreen RHI previews, performance HUD, source-control integration. |
| Networking | **Authoritative server + client-side prediction with rollback**. Full netcode stack. |
| Audio depth | **Steam Audio integrated from MVP1**. HRTF + geometry occlusion + convolution reverb. miniaudio is the playback substrate. |
| Engine extras | **All selected**: Cinematic timeline editor, in-game dev console + Roslyn REPL, Steamworks integration, Save game framework, AI / Navmesh, profiling overlay (Tracy-style), localization + accessibility, crash reporter. |

## 2. High-level architecture

### 2.1 Layer order (bottom-up)

```
+--------------------------------------------------------+
|                    C# Userspace                        |
|  ┌──────────────────────────────────────────────────   |
|  │ Game Code (FLECS components, C# scripts, AI)     │  |
|  ├──────────────────────────────────────────────────┤  |
|  │ Engine Modules (Renderer, Scene, Physics-bind,   │  |
|  │ Audio-bind, Asset, Editor, Networking)           │  |
|  ├──────────────────────────────────────────────────┤  |
|  │ C# Renderer (uses RHI bindings)                  │  |
|  └──────────────────────────────────────────────────┘  |
|                          ▲                             |
|                          │ P/Invoke                    |
|                          ▼                             |
|  ┌──────────────────────────────────────────────────┐  |
|                C Engine Backing                        |
|  ┌──────────────────────────────────────────────────┐  |
|  │ Engine C API (memory, threads, containers, log,  │  |
|  │ RHI, ECS-C-export, physics-C-export, audio...)   │  |
|  ├──────────────────────────────────────────────────┤  |
|  │ RHI (Metal primary; Vulkan stub for later)       │  |
|  ├──────────────────────────────────────────────────┤  |
|  │ FLECS (C)                                        │  |
|  ├──────────────────────────────────────────────────┤  |
|  │ cglm, MikkTSpace, xatlas, tinygltf, basis/univ., │  |
|  │ VHACD, Jolt (C++ compiled to C export), miniaudio│  |
|  └──────────────────────────────────────────────────┘  |
+--------------------------------------------------------+
```

### 2.2 Thread model

- **Game Thread (G)**: ECS progression (FLECS world.progress), gameplay scripts, AI tickers, input polling.
- **Render Thread (R)**: Renderer module pumps the frame graph, issues RHI commands, manages GPU upload ring buffers.
- **Physics Thread (P)**: Jolt simulation step at fixed timestep; interpolated state copied to ECS read-cache on G.
- **Asset IO Thread (I)**: Off-thread PNG/glTF decompression, mesh+uploading to GPU via shared staging ring buffer. Posts Unreal-style progress events to the editor (atomic counter + name). Tiny load work also goes through here.
- **ECS Worker Pool (W₁..Wₙ)**: FLECS job mode dispatches system stages in parallel. Default size = `max(1, hw_concurrency − 2)`.

Cross-thread primitives: SPSC ring buffers for G→R command streams, MPSC queues for I→G/R upload signals, atomic counters for I→editor progress.

### 2.3 Memory model

- **Bump allocators** per subsystem (renderer, physics, audio), reset on appropriate boundary (per-frame for renderer, per-tick for physics).
- **Frame allocator** reset every frame for transient pass data and ECS system scratch.
- **Generational handles** for resources referenced across hot reloads and across threads (mesh handles, texture handles, material handles, audio source handles).
- **No `malloc/free` in hot paths**. Custom heap only where pooling makes sense.

### 2.4 Build system

- **CMake** for the C/C++ side: declares `engine_c` shared library, links FLECS/cglm/MikkTSpace/xatlas/tinygltf/vhacd/Jolt(/miniaudio/Basis Universal), exposes `engine.h`.
- **`dotnet build`**: builds the C# engine (`Engine.dll`), the C# editor (`Editor.dll`), the C# hot-reload target (`Game.dll`).
- **Wrapper script**: A `RunEditor` script (bash/cmd powershell) runs `cmake --build` then `dotnet run --project Editor`.
- **Output layout**: All artifacts under `out/{Debug,Release}/{Arch}/`. Asset authoring and binary cooking placed under `out/{platform}/{config}/cooked/`.

### 2.5 Hot reload pipeline

- **File watcher** on `Game/` source folders.
- **On change**: trigger `dotnet build Game -c {config}` (incremental). Output goes to a deterministic path.
- **Loader**: swap `AssemblyLoadContext` instances — unload old, load new — while persisting entity IDs and component data via managed handles. Resource handles (generational indices) remain stable across reload even if their C# instances move.
- **Renderer hot reload**: same machinery but on the C# renderer assembly. RHI state survives via persistent native handles in C land.

## 3. Tech stack — each dependency and its role

### 3.1 C/C++ libraries (linked into `engine_c`)

| Lib | Role | Notes |
| --- | --- | --- |
| **FLECS** | ECS foundation. Queries, systems, stages, reflection. | C export layer wraps for C#-managed world. |
| **cglm** | Math (vectors, matrices, quats). Pure C, header-mostly, SIMD. | All transforms / camera maths / bounding volumes. |
| **MikkTSpace** | Tangent-space generator for PBR. | Used at asset build time and at runtime if no tangents in source. |
| **tinygltf** | glTF 2.0 source asset loader. | Source-side ingestion; engine references compiled `.mdl`, not raw glTF. |
| **Basis Universal (basisu)** | Master texture compressor. ETC1S/UASTC transcoding to BC/ASTC/ETC2 at upload. | Cook step produces `.ktx2`-like tags. |
| **VHACD** | Convex decomposition of static meshes for physics proxies. | Asset-time use. Outputs into model sidecar meta. |
| **xatlas** | Lightmap UV unwrap + chart packing. | Asset-time use, baked GI keys. |
| **Jolt Physics** | Rigid-body + soft constraint solver. | Wrapped to C export; C# P/Invoke bindings around that. |
| **miniaudio** | Audio engine substrate, decode + playback. | Native C. |
| **Steam Audio** | Spatialisation plugin layer (HRTF, geometry occlusion, reverb). | Plugged into miniaudio buffer post-process. |

### 3.2 C-side

- Modern C17 (no C++ in engine lib). JoltPhysics is C++; isolate behind a thin C export `extern "C" { ... }`.
- Vendored libraries under `third_party/`, fetched by CMake `FetchContent` (or `.cmake` stubs when network is restricted).
- API stability: any C export breaks ABI → bump `ENGINE_ABI_VERSION` in `engine.h`.

### 3.3 C#-side

- **.NET 8+** (modern `LibraryImport`, source-gen marshalling).
- **Flecs.NET** for ECS gameplay.
- **ImGui.NET** for in-game overlays.
- **Avalonia** (with `Avalonia.Diagnostics` + DevTools) for editor chrome.
- **Veldrid** is intentionally **not** used — we want the C RHI to be authoritative; we won't write a C# renderer that uses Veldrid/etc.
- Custom Roslyn wrapper for in-engine dev console REPL and live scripting.

## 4. Asset pipeline

### 4.1 Folder layout

Inspired by Source. Each project has **content folders** with convention:

```
content/
  models/
    <model_name>/
      source.glb           # original glTF (source of truth)
      meta.json            # physics, material overrides, lightmap UV hints
      <model_name>.mdl     # cooked binary (compressed, tagged-format)
  materials/
    <material_name>.mat    # JSON material definition
  textures/
    <tex_name>.ktx2        # Basis-transcoded
  meshes/
    shared/mesh_id.msh     # tag-versioned mesh binary
  sounds/
    <sound>.ogg
    <sound>.audio.src      # source + meta (loop points, mix groups)
  scenes/
    <level>.scene.json     # JSON scene referencing compiled assets
  scripts/
    <script_type>.cs       # C# scripts (hot-reload)
```

### 4.2 Format philosophy — tag-format dual backends

A tag-encoded format lets the engine identify what compression/encoding scheme is in use and decode accordingly, even across engine upgrades:

- **Mesh tags**: `MESH_V1`, `MESH_V1_LZ4`, `MESH_V1_ZSTD`, `MESH_V1_LZ4_F16` ...
- **Texture tags**: `KTX2_BASIS_ETC1S`, `KTX2_BASIS_UASTC`, `KTX2_RAW`, `KTX2_RAW_BC7`.
- **Audio tags**: `AUDIO_OGG`, `AUDIO_WAV`, `AUDIO_OPUS`.
- **Sidecar `meta.json`**: collider shapes, material assignments per submesh, lightmap UV chart assignments, mass, friction, restitution.

Tag reader in C knows all current tags. A migration system writes a sidecar `cook_log.json` describing the cook chain so rollback or regen is reproducible.

### 4.3 Cook step

Standalone CLI: `engine_cook [project_dir]`. Operations:

1. **Models**: ingest `source.glb` → strip/compress → compute MikkTSpace tangents → emit `.msh` + `.mdl` sidecar.
2. **Physics**: VHACD convex decomposition → write convex hull payload into `.mdl` sidecar meta.
3. **Lightmap**: xatlas UV unwrap → write UV chart payload into `.mdl` sidecar + `.lmap.uv` payload.
4. **Textures**: Basis Universal compress → write `.ktx2`.
5. **Audio**: passthrough with mix-group + loop-point extraction → `.audio`.

### 4.4 Scenes — JSON

Source-of-truth for level layout. References compiled assets by path (relative to project root) and tag.

```json
{
  "name": "test_scene",
  "version": "1.0",
  "ambient_color": [0.5, 0.6, 0.7],
  "ambient_intensity": 1.0,
  "entities": [
    {
      "name": "test_entity",
      "position": [0, 0, 0],
      "rotation": [0, 0, 0, 1],
      "scale":    [1, 1, 1],
      "components": [
        { "type": "mesh",     "mesh": "models/test/test.mdl", "material_override": "materials/test.mat" },
        { "type": "transform" },
        { "type": "physics_static" },
        { "type": "script", "script": "scripts/spin.cs" }
      ]
    }
  ]
}
```

### 4.5 Animations (later)

`Skeleton` component, `AnimationClip` resource, blends and state machines are intentionally deferred per user instruction. Scaffolding for `skeleton` and `animation_clip` components will exist but remain empty until the animation phase. Use glTF skin data only to populate initial skeleton; full playback system is post-MVP1.

## 5. Renderer

### 5.1 Architecture

- **C RHI** — minimal, GPU-API-abstracted. Targets **Metal first**, **Vulkan later**. Exposes a C API in `engine.h`.
- **C# Renderer** — uses the C RHI bindings. Implements Forward+ pipeline, PBR shading, mesh-shader culling, light clustering, post-processing (tonemap, SMAA, bloom optional, optional screen-space AO).
- **Frame-graph**-style pass scheduler — list of passes with resource declarations; barrier planning lives in the C RHI (analogous to Vulkan/Metal best practices).

### 5.2 Pipeline stages

1. **Mesh shader (object cull + LOD selection)** — per-task cull into threadgroups, per-mesh LOD index → output draw-list.
2. **Clustered light culling** — compute shader builds tile/cluster light lists.
3. **Forward+ opaque pass** — vertex + mesh-shader dispatched with light clusters.
4. **Forward transparency pass** — sort required here, no clusters interaction (single light baked into vertex).
5. **Particles** — separate pass with depth read for soft particle intersection.
6. **Post-processing chain** — SMAA edge detect + blend → bloom (optional) → ACES → tonemap → present.

### 5.3 Material system (Disney PBR)

- Single shading function with: base color, metallic, roughness, normal, occlusion, emissive, sheen, clearcoat, transmission, anisotropy.
- Feature bitset toggles cheaper paths.
- Material authored via JSON `.mat` referenced by scene or by mesh slot.
- Editor can live-edit a `.mat` and see it in a model viewer.

### 5.4 Water

- **Indoor-scene scale**: height-map-based, single reflection plane per scene.
- **Flow maps**: scroll UVs in tangent space.
- **Reflections**: SSR (one-bounce planar) backed by baked cubemap fallback.
- **Absorption**: depth-based color absorption.
- **Soft refraction**: scene depth sample + distortion offset.
- **Ray-tracing optional**: Metal RT support flag — when available, refraction becomes true ray-traced thin glass. Falls back gracefully.

### 5.5 Particles

- Editor lives in the particle editor suite.
- CPU particles for low/med counts, GPU compute particles for high counts.
- Soft-particle depth-buffer intersection.
- Sprite + ribbon + mesh emitter types.
- Curve-driven modules (size over life, color over life, etc.).

## 6. RHI — concrete API surface

Stable C ABI in `engine.h`:

```c
typedef struct RhiDevice RhiDevice;
typedef struct RhiSwapchain RhiSwapchain;
typedef struct RhiBuffer RhiBuffer;
typedef struct RhiTexture RhiTexture;
typedef struct RhiPipeline RhiPipeline;
typedef struct RhiCommandList RhiCommandList;
typedef struct RhiShader RhiShader;

ENGINE_API RhiDevice*    rhi_create_device(void);
ENGINE_API void          rhi_destroy_device(RhiDevice*);

ENGINE_API RhiSwapchain* rhi_create_swapchain(RhiDevice*, void* ns_window);
ENGINE_API uint32_t      rhi_acquire_next_image(RhiSwapchain*);
ENGINE_API void          rhi_present(RhiSwapchain*);

ENGINE_API RhiBuffer*    rhi_create_buffer(RhiDevice*, const RhiBufferDesc*);
ENGINE_API RhiTexture*   rhi_create_texture(RhiDevice*, const RhiTextureDesc*);
ENGINE_API RhiPipeline*  rhi_create_pipeline(RhiDevice*, const RhiPipelineDesc*);
ENGINE_API RhiShader*    rhi_compile_shader(RhiDevice*, const RhiShaderDesc*);
ENGINE_API void          rhi_destroy_buffer(RhiBuffer*);
// ...

ENGINE_API RhiCommandList* rhi_begin_cmdlist(RhiDevice*, RhiSwapchain*);
ENGINE_API void            rhi_cmd_bind_pipeline(RhiCommandList*, RhiPipeline*);
ENGINE_API void            rhi_cmd_set_buffer(RhiCommandList*, uint32_t slot, RhiBuffer*);
ENGINE_API void            rhi_cmd_draw_mesh(RhiCommandList*, uint32_t vertex_count, uint32_t instance_count, uint32_t first_vertex);
ENGINE_API void            rhi_cmd_dispatch(RhiCommandList*, uint32_t groups_x, uint32_t groups_y, uint32_t groups_z);
ENGINE_API void            rhi_end_cmdlist(RhiCommandList*);
```

**Memory model in RHI**:
- Persistent buffers/textures/pipelines: created once during init / asset load, never per-frame.
- Triple-buffered uniform/scene data ring buffers for CPU→GPU per-frame upload.
- Modern Metal best practices: persistent `MTLDevice` / `MTLCommandQueue`, immutable resources, argument buffers for resource binding efficiency.
- Stack-allocated per-pass transient data via the C frame allocator.

## 8. Editor — Avalonia + ImGui

### 8.1 Window architecture

- One **main editor window** hosts a docked Avalonia layout with:
  - **Viewport panel** (renders engine RHI output to an embedded `RenderTargetBitmap`-passed texture or shared `MTLTexture` handle).
  - **Asset browser** (tree + thumbnail grid; thumbnails rendered via offscreen RHI capture).
  - **Hierarchy panel** (FLECS entity tree).
  - **Inspector panel** (selected entity or asset).
  - **Console panel** (engine log + dev console).
  - **Profiler panel** (per-system timing, GPU markers).
  - **Timeline / cinematic panel** (when a cinematic asset is open).
  - **Localization panel** (when a localization table is open).
- Docking layout persisted per user + per asset type; themes skinnable.
- Native menu bar with File / Edit / Window / Tools / Help, like Source/Unreal.

### 8.2 ImGui overlay

- In-viewport ImGui draws **scene controls** (gizmos, gizmo handles via ImGuizmo), material viewers, particle preview, model preview.
- ImGui input lifter pulls from Avalonia's hosted window input model.
- Clipping rectangle mapping via Avalonia host control.

## 9. Physics

- **Jolt Physics** as solver (broadphase + narrowphase + constraint solver).
- C wrapper exposes C entrypoints in C-style ~ `physics_body_create`, `physics_step(world, dt)`, etc.
- C# binding wraps C wrapper, not C++.
- VHACD pre-baked convex hulls as referenced proxies from `.mdl` sidecar.
- Fixed timestep **60 Hz** (or configurable), with interpolation for state presentation.
- Physics thread runs independently; ECS physics components read interpolated state on Game Thread.

## 10. Audio

- **miniaudio** is decoder + playback substrate (stereo + 5.1).
- **Steam Audio** integrated from MVP1:
  - HRTF spatialisation for 3D sounds.
  - Geometry-based occlusion against scene geometry (Steam Audio mesh built from cooked scene geometry).
  - Convolution reverb (Steam Audio baked reverb presets per scene).
- **Audio scene** = Steam Audio baked sub-scene + miniaudio voice graph.
- Mixing buses: master, music, SFX, voice, ambience. Volume curves per bus.

## 11. ECS — FLECS integration

- One FLECS world per game instance.
- **Core component schema** (`Transform`, `Visibility`, `Tag`, `MeshHandle`, `MaterialHandle`, `PhysicsBodyHandle`, `AudioEmitterHandle`, `ScriptRef`).
- **Game components** defined by user code in C# (registered via Flecs.NET reflection).
- **C systems** (GPU upload, asset streaming) run as FLECS systems on Render/IO threads using FLECS job mode.
- Components can hold **script references** — a path to a C# class implementing a known interface (`IComponentLogic`). On entity tick the engine queues a managed call dispatch.

---

## 12. Networking

- **Authoritative server model** with rollback:
  - Server simulates physics + authoritative gameplay at fixed tick.
  - Client receives state delta, predicts locally, rolls back on correction.
  - ECS-native state replication: components opt in via `[Replicated]` attribute; snapshot delta encoded with binary tag-format.
- **Transport**: ENet (UDP with reliability layers) for MVP1.
- **Rollback**: per-tick snapshot of relevant components; resize-friendly.

---

## 13. Save game framework

- **Snapshot diff format**: snapshot entire ECS world state; diff against last loaded snapshot to minimize payload.
- **Stable component handles** across sessions (component GUIDs baked at compile time).
- **Field-level dirty tracking** optional. Default = full-blob snapshot.
- Save format = containerized JSON metadata + binary BLOB for entity state.

## 20. Critical "first steps" — in order

Per user's explicit instruction — **logging first, then Avalonia, then RHI**.

### 20.1 Step 1 — Logging system (highest priority)

- C-side: lock-free ring-buffer logger with severity levels (trace, debug, info, warn, error, fatal). Multi-sink: stdout, file (rolling), in-memory ring memory-buffer for UI.
- C# bindings: dump captured logs to Avalonia console panel + accept new log messages from C#.
- Crash-on-error behavior: opt-out default; in dev builds, fatal → dump diagnostic + break-into-debugger if available.

### 20.2 Step 2 — Avalonia setup

- New project: `Editor/Avalonia.Editor/`.
- Avalonia app shell + dock layout + theme.
- Empty panels stub-out for Viewport, AssetBrowser, Hierarchy, Inspector, Console.
- Embed a native RHI surface later (post-RHI).

### 20.3 Step 3 — RHI primary focus (Metal)

- C-side: implement the RHI surface in C with Metal backend (Apple Silicon baseline). Persistent objects, immutable resources, triple-buffered ring buffers.
- Stub Vulkan backend definition only — implement when Vulkan later.

### 20.4 Subsequent steps (provisional order)

1. ECS bootstrap + Flecs.NET wrapper + a window-running "Hello Triangle" using C# Renderer.
2. Asset foundation: scenes JSON parser; stub material/mesh handles (no asset yet).
3. Asset pipeline deep: model/mdl format, mesh tags, Basis Universal textures.
4. PBR shading (Disney Principled), mesh-shader driven Forward+.
5. Clustered light culling + MSAA 4× + SMAA.
6. Material editor + Model viewer MVP.
7. Editor full set (Level + Particle + Model + Material).
8. Physics (Jolt + VHACD integration in cook step).
9. Audio (miniaudio + Steam Audio).
10. Networking (authoritative + rollback).
11. AI / Nav / Cinematic / Save / Steamworks / Localization / Crash reporter / Profiling overlay / Dev console.
12. **Expansion phase (later)**: Vulkan port + terrain system.

(Terrain explicitly deferred per user. Animation likewise deferred but with skeleton/animation components scaffolded.)

## 21. Risks & mitigations

| Risk | Mitigation |
| --- | --- |
| **Flecs.NET version drift** (community) | Lock to specific commit, vendor in repo, isolate behind engine wrapper. |
| **C ABI drift breaking hot reload** | Version every C export; bump `ENGINE_ABI_VERSION` on breaking changes; bind with `LibraryImport` (compile-time). |
| **Metal-only renderer vs Vulkan later** | Define RHI C surface up front to abstract; Vulkan backend reuses the same API. Avoid Metal-only assumptions in high-level render passes. |
| **Stack-alloc leak on long-lived handles** | All cross-frame data goes through handle indices, never raw pointers. |
| **Steam Audio on Apple Silicon quirks** | Test Steam Audio's macOS support early; if blocked, integrate geometry-based alternative. |
| **Mesh-shader requirements gating older hardware** | Document Metal 3 / Vulkan 1.3 baseline; provide soft fallback to vertex-shader path. |
| **Hot reload state corruption on schema change** | Snapshot ECS state pre-reload; user prompt to migrate if schema incompatible, else auto-recreate. |
| **Avalonia + native RHI surface embed** | Use shared MTLTexture handle, copy to `Avalonia.Media.Imaging.Bitmap` for MVP1; investigate GPU-direct embed later. |

## 25. Project descriptor — `.eeproj` (LOCKED)

### 25.1 Purpose

`.eeproj` is the project's source-of-truth descriptor that lives at the root of every project folder. Conceptually it fuses Unreal's `.uproject`, Unity's `ProjectSettings.asset`, Godot's `project.godot`, and Source's `gameinfo.txt`: it declares identity, layout, default subsystems, addon list, shippable scenes, and source-control behavior.

It is **not** a runtime asset: the engine reads it at editor startup, cook time, and at game launch in dev mode. Otherwise the engine boots with no `.eeproj` awareness.

### 25.2 Format — split directory of JSON sub-files

`.eeproj` is **not a single file**. It is a directory: `<project>/.eeproj/`, containing seven JSON sub-files. Splitting reduces git merge conflicts (subsystem changes can't collide) and lets each module own its config slice.

| File | Authoritative for |
| --- | --- |
| `project.json` | Identity, save folder, Steam app id, network tick rate, external mounts. |
| `scenes.json` | Shippable scenes list, startup scene, main menu scene. |
| `modules.json` | Renderer, physics, audio subsystem defaults. |
| `addons.json` | Opt-in addon list (Steam Audio, Steamworks, team extensions). |
| `input.json` | Action/axis mappings and default bindings. |
| `locales.json` | Default locale, supported locales, fallback chain. |
| `editor.local.json` | Per-workstation state (window layouts, recent scenes, Tracy URI). **Gitignored.** |

### 25.3 Per-file schema versioning

Each sub-file carries its own `"version": 1` field. Subsystem decoupling is intentional: if `render.json` (inside `modules.json`) gains a Forward+ cluster-config field in v2 but `physics.json` stays at v1, only the renderer module handles its own upgrade. No whole-project migration ceremony.

### 25.4 Atomic writes

The editor never rewrites a `.eeproj/*.json` file directly. Writes go to a sibling `.tmp`, and the editor performs an atomic `rename()` to commit. The same pattern protects against process kills mid-save.

### 25.5 Mounts vs addons

These are distinct concepts that get confused easily. The rule:

- **External mount** = a **content-only** VFS folder exposed read-only. It can contribute scene files, models, textures, sounds — but the assets must still be **listed in the project's `scenes.json`** if they are to ship with a build. Mounts do not auto-inject into `scenes.json`.
- **Addon** = a **code-level** engine plugin. Activated before the game's hot-reload assembly loads. Addons may optionally register a hidden VFS mount (e.g., Steam Audio's bundled IR files).

| | Mount | Addon |
| --- | --- | --- |
| Content | Yes | Optional hidden content |
| Code hooks | No | Yes |
| Declared in | `project.json#external_mounts` | `addons.json#enabled` |
| Identifier | path (local or `url@commit`) | reverse-DNS + semver |
| Hot-reload | YES — content can hot-reload live | NO — requires editor restart |

### 25.6 `project.json` — example

```json
{
  "version": 1,
  "guid": "8c3e62b1-9a4e-4f01-8e6c-7f9f1d3a8b71",
  "name": "Test Game",
  "company": "Test Studio",
  "steam_app_id": 480,
  "save_game_folder_name": "TestGame",
  "network_tick_rate": 60,
  "external_mounts": [
    { "path": "../SharedBaseGame",  "priority": 1, "readonly": true },
    { "url":  "https://git.team/shared-base.git", "commit": "a1b2c3d4", "priority": 0 }
  ]
}
```

**Mount priority rules**: earlier items in `external_mounts` win. The project itself has implicit priority = `+inf`. This is identical to Source's `gameinfo.txt` ordering semantics.

**Mount version pinning**: local paths rely on git submodules. Remote mounts have an explicit `commit` field; the engine refuses to silently update to a different `HEAD`.

### 25.7 `scenes.json` — example

```json
{
  "version": 1,
  "startup_scene":   "Content/scenes/main_menu.scene.json",
  "main_menu_scene": "Content/scenes/main_menu.scene.json",
  "cooked_scenes": [
    "Content/scenes/main_menu.scene.json",
    "Content/scenes/level01.scene.json",
    "Content/scenes/level02.scene.json"
  ]
}
```

**Scene list semantics**:

- The cook step processes **only** entries in `cooked_scenes[]`. Unreferenced/unlisted scenes are stripped from cooked output by default.
- The editor watches `scenes.json` for changes (external `git pull` updates it instantly).
- The editor surfaces a badge *"Not in build list"* on scenes that exist but are not enumerated. Dragging a scene into the project just creates the file — adding it to the build is a separate explicit action.
- Unlisted scenes can still be opened at runtime via the dev console or `Engine.Scene.OpenScene(path)` (provided they are cooked — dev-only escape hatch).
- `startup_scene` is the boot entry; `main_menu_scene` is the menu boot fallback if no save game exists.

### 25.8 `modules.json` — example

```json
{
  "version": 1,
  "renderer": {
    "default_pipeline":   "mesh_shader_forward_plus",
    "msaa":               4,
    "post_aa":            "smaa",
    "tonemapper":         "aces_filmic",
    "hdr":                true,
    "shadow_resolution":  2048,
    "lightmap_format":    "rg16f"
  },
  "physics": {
    "engine":             "jolt",
    "fixed_timestep_hz":  60,
    "max_substeps":       4,
    "gravity":            [0, -9.81, 0],
    "default_layer_count": 32
  },
  "audio": {
    "backend":  "miniaudio_steam_audio",
    "hrtf":     true,
    "buses":    ["Master", "Music", "SFX", "Voice", "Ambience"],
    "default_master_volume": 0.85
  }
}
```

These are **project defaults**, overridable per scene. Default-per-project keeps scenes portable when copied between projects.

### 25.9 `addons.json` — example

```json
{
  "version": 1,
  "enabled": [
    { "id": "core.steamworks",  "version": ">=1.0.0" },
    { "id": "core.steam_audio", "version": "1.2.0"      },
    { "id": "team.weaponmod",   "version": "0.2.1", "path": "Addons/WeaponMod/" }
  ]
}
```

**Addon identifier format**: reverse-DNS scope (`com.author.plugin`) or short scoped (`core.*`, `team.*`) + semver. Paths are optionally local.

### 25.10 `input.json` — example

```json
{
  "version": 1,
  "actions": {
    "jump":         { "primary": "Space",         "gamepad": "GamepadSouth" },
    "move_forward": { "primary": "KeyW",          "axis": { "value":  1 } },
    "move_back":    { "primary": "KeyS",          "axis": { "value": -1 } },
    "fire_primary": { "primary": "MouseLeftBtn", "gamepad": "GamepadRight" }
  }
}
```

`input.json` becomes the **canonical** name → binding mapping that user code references by name (`input.IsPressed("jump")`). Bindings can be remapped at runtime and the editor persists changes here.

### 25.11 `locales.json` — example

```json
{
  "version": 1,
  "default_locale":    "en-US",
  "supported_locales": ["en-US", "ja-JP", "fr-FR"],
  "fallback_chain":    ["en-US"]
}
```

`fallback_chain` is searched in order if a key is missing in the active locale.

### 25.12 `editor.local.json` (gitignored)

```json
{
  "version": 1,
  "recent_scenes":     ["Content/scenes/main_menu.scene.json"],
  "window_layouts":    [...],
  "tracy_uri":         "tcp://127.0.0.1:8120",
  "theme":             "dark",
  "last_accessed":     "2026-06-25T12:34:56Z"
}
```

Per-workstation state. Lives in `.eeproj/editor.local.json` so it can benefit from the same schema versioning, but **must be added to `.gitignore`**.

### 25.13 Source-control behavior

- `.eeproj/project.json`, `scenes.json`, `modules.json`, `addons.json`, `input.json`, `locales.json` are **text-tracked**. They are small and diff-friendly. **Do NOT place `.eeproj/*.json` files in Git LFS.**
- `editor.local.json` is **gitignored** entirely (user-local, not project-wide).
- `external_mounts` that point to git URLs should be cloned as **git submodules** for local path-mounts to behave correctly.
- **No lock-on-edit semantics** for any `.eeproj/*.json`. Merges are normal text merges. Lock-on-edit applies only to the binary cooked assets that the project references (`.mdl`, `.ktx2`, `.audio`).
